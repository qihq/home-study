# 家庭学习助手详细设计规格

**版本：** 1.0  
**日期：** 2026-07-12  
**部署目标：** Synology DS918+（DSM 7 / Container Manager）  
**主要终端：** iPhone、iPad、桌面浏览器；移动端支持 Safari 或添加到主屏幕后的 PWA  
**文档状态：** 待用户最终审阅  

## 1. 文档目的

本文定义一个部署在群晖 NAS 上的家庭学习助手。第一版包含三个互相关联、但边界清晰的模块：

1. 阅读打卡：在 iPhone/iPad 网页中分别录制中文、英文阅读视频，录制期间分片上传 NAS。
2. 单词默写：从图片、粘贴文本、Excel、Word 或 PDF 导入单词，使用标准美式 AI 语音进行手动控制的默写。
3. 学习统计：统计阅读完成情况、时长、连续天数、默写正确率和错词历史。

本文面向后续编码执行者。未明确写入本文的功能不属于第一版交付范围。

## 2. 已确认需求

### 2.1 阅读视频

- 中文阅读和英文阅读每天分别录制为两个视频。
- 从应用网页直接录像，不先使用系统相机生成本地完整视频。
- 录像过程中将小片段持续上传 NAS，降低手机长期占用的本地空间。
- NAS 永久保留完整源视频，不执行自动过期删除。
- NAS 自动生成 720p 压缩版，格式为 MP4（H.264 + AAC）。
- 用户在 iPhone/iPad 上手动下载压缩版，并通过 iOS 分享菜单选择“存储视频”。
- 同一天、同一阅读类型允许录制多次，只选择其中一次作为正式打卡。
- 外网域名、证书、端口映射和群晖反向代理由用户自行配置；应用必须能在标准 HTTPS 反向代理后工作。

### 2.2 单词默写

- 支持拍照识别、复制粘贴、Excel、Word、PDF 五种导入入口。
- 导入后必须显示可编辑的确认清单，系统不得在未经确认时直接开始默写。
- 默认使用标准美式发音。
- 默认使用 Xiaomi MiMo TTS；语音提供商必须采用可替换适配器设计。
- OpenCode Go 的 MiMo 文本模型可用于可选的格式整理和疑似错误提示，但 OpenCode Go 的聊天接口不能作为 TTS 接口。
- 单词音频生成后永久缓存在 NAS，重复使用时不再次请求 TTS。
- 默写默认按导入顺序，开始前可以切换随机顺序。
- 由家长手动触发播放、重播、下一个、显示答案，并人工标记正确或错误。
- 不做手写识别和自动批改。

### 2.3 统计

- 阅读统计包含周/月日历、中文和英文完成率、累计时长、当前连续天数、最长连续天数。
- 默写统计包含每次总数、正确数、正确率、错词、周/月趋势、单词累计错误次数。
- 错词可一键生成新的复习列表。
- 第一版只提供一个家庭管理员账号和一个默认孩子档案，但数据库结构支持未来增加多个孩子。
- UI 必须三端适配：iPhone、iPad 和桌面浏览器。录制主流程优先保障 iPhone/iPad Safari；视频库、统计、设置和故障处理在桌面端提供更高信息密度的管理布局。

## 3. 关键定义和限制

### 3.1 “源视频”的定义

源视频是浏览器通过 `MediaRecorder` 产生、上传到 NAS 后未经二次压缩的录制内容。它保留浏览器实际采集到的分辨率、码率和编码格式，但不是相机传感器的无损原片，也不等同于使用 iOS 系统相机得到的最高质量文件。

如果设备、浏览器或网络条件不允许指定的理想参数，应用可退回设备支持的参数，但必须在录制前显示实际分辨率和预计码率。

### 3.2 iOS 限制

- 录像期间页面必须保持前台；锁屏、切换应用、系统回收 Safari 页面都可能中断录像。
- PWA 不能在用户无操作的情况下把视频静默写入“照片”。转码完成后，用户仍需点击“下载/保存”，再使用 iOS 系统界面保存视频。
- 断网时新片段只能临时存放在 IndexedDB。Safari 的可用存储额度由系统动态决定，不能承诺无限离线录制。
- 应用目标是避免保存一份完整视频到手机，而不是保证录制时手机完全不使用临时空间。

### 3.3 外部访问假设

应用只假设最终入口为 HTTPS，例如 `https://study.example.com`。公网 IP、DNS、证书、反向代理和防火墙不在本项目范围内。摄像头和麦克风权限依赖安全上下文，因此生产环境不得使用普通 HTTP。

## 4. 第一版范围边界

### 4.1 包含

- 单管理员登录。
- 一个默认孩子档案。
- PWA 首页、录制、视频库、单词导入、默写、错词本、统计和管理页面。
- 分片上传、断点续传、源视频组装、720p 转码、失败重试。
- MiMo TTS 适配器和音频缓存。
- 本地文档解析、OCR、可选 AI 整理。
- 数据导出、数据库备份说明、任务与存储状态查看。

### 4.2 不包含

- 原生 iOS/iPadOS 应用。
- 静默保存到系统相册。
- iOS 后台持续录像或后台无限上传。
- 自动识别孩子朗读内容、发音评分或阅读质量分析。
- 手写单词识别和自动批改。
- 推送通知、短信、微信通知。
- 多管理员、复杂家庭权限和公开分享链接。
- 自动删除已完成的源视频。
- 公网网络和群晖反向代理的自动配置。

## 5. 推荐技术架构

### 5.1 技术栈

为适配 DS918+ 的资源和维护成本，推荐：

- 前端：React + TypeScript + Vite，使用 PWA manifest 和 service worker。
- 后端：Python 3.12 + FastAPI。
- 数据库：SQLite，启用 WAL、外键和定期在线备份。
- 后台任务：同一后端代码库的独立 worker 进程，通过 SQLite `jobs` 表领取任务。
- 视频：FFmpeg/ffprobe，优先使用 Intel VAAPI 硬件编码，失败时回退 `libx264`。
- OCR/文件解析：Tesseract OCR、Pillow、pypdf、python-docx、openpyxl；扫描 PDF 需要 Poppler 渲染后 OCR。
- 部署：Docker Compose，由群晖 Container Manager 运行。

第一版不引入 Redis、消息队列或独立 PostgreSQL。转码并发固定为 1，OCR/TTS 后台任务也限制并发，避免 DS918+ 被打满。

### 5.2 容器组成

```text
reverse proxy (用户维护)
        |
        v
web/api container -------- SQLite + media volume
        |
        +---------------> worker container
                              |-- FFmpeg / ffprobe
                              |-- OCR / document parser
                              +-- MiMo API
```

建议使用两个应用容器：

- `app`：提供静态 PWA、REST API、认证和下载。
- `worker`：处理组装、转码、OCR、TTS、清理废弃临时片段和统计修复。

两个容器使用同一个镜像、不同启动命令，共享 SQLite 和媒体目录。`worker` 独占视频转码任务锁。

### 5.3 NAS 卷目录

容器内部建议统一挂载为：

```text
/data/
  app.db
  backups/
  uploads/
    recordings/<recording-id>/chunks/
    imports/<import-id>/
  videos/<child-slug>/<yyyy>/<mm>/<dd>/<chinese|english>/<recording-id>/
    source.<ext>
    720p.mp4
    metadata.json
  tts/en-US/<provider>/<hash-prefix>/<hash>.mp3
  logs/
```

群晖实际共享目录可为 `/volume1/family-learning`，但应用不得硬编码宿主机路径。

## 6. 页面与用户流程

### 6.1 登录页

- 输入管理员账号和密码。
- 登录成功后使用 HttpOnly、Secure、SameSite=Lax 的会话 Cookie。
- 连续失败需要限速；不得在日志中记录密码或 API Key。
- 登录卡片在 iPhone 上全宽且保留安全区边距，在 iPad/桌面上限制最大宽度并居中。

### 6.1.1 三端响应式布局规则

采用同一套 PWA，不维护独立的移动/平板/桌面代码分支。以可用内容宽度，而非设备型号判断布局：

| 视口 | 主要布局 | 导航 | 重点要求 |
| --- | --- | --- | --- |
| 小于 768px（iPhone） | 单列、底部主导航 | 底部导航 + 页面顶部返回 | 44px 以上点击区域、适配刘海和底部安全区；录制控制固定在底部；统计图表纵向排列。 |
| 768px 至 1199px（iPad） | 双列或宽单列 | 顶部栏或可折叠侧栏 | 录制预览和上传信息可并排；单词编辑与默写控制可并列；横竖屏均不得溢出。 |
| 1200px 及以上（桌面） | 内容区 + 持久侧栏/顶部导航 | 持久导航 | 视频库、任务队列、统计表格可使用多列和更完整筛选；仍限制正文最大宽度，避免超宽行。 |

所有功能必须三端可访问，不得把管理、统计或恢复入口只放在桌面端。桌面端不承诺摄像头驱动的录制质量；录制页在桌面显示兼容性提示，但在浏览器支持时仍可使用。

### 6.2 首页

首页优先展示当天任务：

- 中文阅读：未完成、处理中或已完成；显示正式视频时长。
- 英文阅读：未完成、处理中或已完成；显示正式视频时长。
- 主按钮：“开始中文阅读”“开始英文阅读”“开始默写”。
- 本周概览：阅读总完成率、当前连续打卡天数、最近一次默写正确率。
- 未完成恢复卡片：存在未结束上传时，显示“继续上传/检查录制/放弃”。
- 后台任务提示：转码或 TTS 失败时显示可操作的重试入口。

### 6.3 录制前页

- 选择中文阅读或英文阅读。若从对应首页按钮进入则预选。
- 显示孩子、打卡日期、摄像头预览、麦克风状态和网络状态。
- 默认使用前置摄像头；允许切换前/后摄像头。
- 推荐采集参数：优先 1920x1080、30fps；设备不支持时退到 1280x720、30fps。
- 录制前通过运行时能力检测选择 MIME 类型，不能只依赖 User-Agent。首选 Safari 支持的 H.264/AAC MP4；如果只能输出其他格式，服务端仍需接收并标准化。
- 显示实际获得的轨道分辨率、估计上传速率、当前剩余缓冲能力提示。
- 用户必须确认“录制期间保持本页前台”。

### 6.4 录制中页

- 大号计时器、红色录制状态、中文/英文类型、实时预览。
- 主要操作只有“结束录制”；次要操作为静音提示和取消。
- 每 4 秒触发一个 `MediaRecorder` 数据片段。实现可根据真机测试在 2–5 秒间调整，但不得超过 5 秒。
- 片段先写入 IndexedDB，再上传；服务端确认校验通过后才从 IndexedDB 删除。
- 页面显示：已录时长、已上传时长、待上传大小、当前网络状态。
- 网络断开时继续录制并缓存，同时明显提示“离线缓存正在增长”。
- 待上传缓存超过 150 MB 时黄色预警；超过 250 MB 或浏览器存储写入失败时必须自动安全结束录制，避免页面被系统强制终止。阈值需要做成配置项。
- `pagehide`、`visibilitychange`、媒体轨道结束、麦克风/摄像头权限被撤回时，立即持久化录制状态并尝试停止录制。

### 6.5 结束与上传确认页

- 用户结束录制后，录制状态变为“等待片段上传完成”。
- 页面不得在最后片段获得服务端确认前显示“源视频已保存”。
- 全部片段上传后调用完成接口，服务端校验序号连续、大小和哈希一致，再创建组装任务。
- 页面可关闭；稍后回到应用继续查看组装和转码进度。
- 源视频通过 ffprobe 验证可读取、存在视频和音频轨道、时长合理后，记录才进入 `source_ready`。
- 第一次成功的同日同类型视频自动成为正式打卡。后续视频保存为备用，不自动替换正式视频。

### 6.6 视频库与详情页

- 按日期、中文/英文、状态筛选。
- 显示源视频状态、720p 状态、时长、大小、创建时间和是否为正式打卡。
- 同日多次录制允许切换正式视频；切换后统计立即重新计算。
- 转码失败时显示错误摘要和“重新转码”。
- 720p 完成后提供“在 iPhone/iPad 保存”按钮。
- 保存按钮使用带认证 Cookie 的直接文件响应。优先让 Safari 打开系统视频预览，再提示使用分享菜单“存储视频”；不得把大视频完整读入前端 JavaScript 内存。
- 第一版不提供已完成源视频删除按钮。管理员若在 NAS 文件管理器手动删除文件，应用检测后标记为 `missing`，不得悄悄把记录当作正常。

### 6.7 单词导入页

提供以下入口：

1. 拍照/选择图片：支持 JPG、PNG、HEIC。HEIC 在服务端转换后 OCR。
2. 粘贴文本：支持换行、空格、逗号、分号和编号列表。
3. Excel：支持 `.xlsx`，扫描所有非空单元格，优先识别明确命名为 `word`、`单词`、`英文` 的列。
4. Word：支持 `.docx` 的正文和表格。
5. PDF：优先提取文本层；无有效文本层时逐页渲染并 OCR。

第一版不承诺兼容旧版 `.xls` 和 `.doc`。上传时应给出明确转换提示，不得错误显示为“系统故障”。

导入流程分为四步：上传/输入、解析、确认、生成语音。解析结果必须保留原始文本和来源定位，便于用户核对。

### 6.8 单词确认页

- 列表逐项显示候选单词/短语、来源和疑似问题。
- 支持编辑、删除、增补、拖动排序和批量去重。
- 允许英文单词、包含空格的短语、连字符和撇号；单项长度限制为 1–80 个字符。
- 规范化只用于比较和缓存：去除首尾空白、合并连续空格、Unicode 标准化；不得擅自改变用户确认的显示拼写。
- 本地规则先标记含数字、异常符号或超长项目。
- 可选调用 OpenCode Go 的 `MiMo-V2.5` 进行格式整理和疑似拼写错误提示。AI 只能返回建议，不得自动替换。
- 用户点击“确认单词本”后固化列表版本，并创建缺失音频的 TTS 任务。

### 6.9 单词本详情页

- 显示单词数量、语音就绪数量、创建日期和来源文件。
- 单个单词可试听、编辑和重新生成语音。
- 编辑已确认单词时生成新版本，历史默写记录继续引用原版本，避免历史答案变化。
- 所有音频就绪前允许进入默写，但未就绪项必须明确标记并禁止无声跳过。

### 6.10 默写页

开始前：

- 选择单词本或错词复习列表。
- 选择“按导入顺序”或“随机顺序”，默认按导入顺序。
- 随机模式在开始时生成并保存一个固定顺序；刷新或恢复会话时顺序不变。

默写中：

- 默认隐藏英文拼写。
- 显示进度，例如 `7 / 20`，但不显示当前答案。
- “播放”按钮由家长手动触发；支持任意次数“重播”。
- “下一个”只切换项目，不自动倒计时。
- “显示答案”后显示拼写，随后必须由家长选择“正确”或“错误”。
- 未评分前进入下一项时需要轻量确认，避免漏记。
- 每次播放、显示答案、评分都即时保存，页面刷新后可以继续。

结束后：

- 显示总数、正确数、错误数、正确率和错词列表。
- 提供“只复习本次错词”和“加入错词本”。
- 允许修改误点的评分；修改必须同步更新统计汇总。

### 6.11 统计页

提供三个视图：阅读、默写、错词。

阅读视图：

- 周/月日历每天分别显示中文和英文完成状态。
- 中文累计时长、英文累计时长、两者总时长。
- 中文完成率、英文完成率和综合完成率。
- 当前连续天数和最长连续天数。

默写视图：

- 每次默写的日期、单词本、总数、正确数、正确率。
- 周/月正确率趋势。
- 总练习次数、累计单词评分次数和去重后的练习词数。

错词视图：

- 按累计错误次数、最近错误时间排序。
- 显示每个单词的正确次数、错误次数、最近结果。
- 选择多个错词生成复习列表。

### 6.12 管理页

- 修改管理员密码。
- 配置孩子显示名称和时区，默认 `Asia/Shanghai`。
- 配置 MiMo TTS API Key、模型、音色、语速和 API Base URL。
- 配置可选 OpenCode Go Key 和模型。
- TTS 配置必须提供“生成并试听测试单词”的验证按钮。
- 显示 NAS 媒体用量、数据库大小、未完成上传、任务队列和失败任务。
- 导出学习数据为 JSON/CSV；导出不默认打包视频文件。
- 触发 SQLite 在线备份。

## 7. 视频数据流与状态机

### 7.1 录制会话状态

```text
created
  -> recording
  -> uploading
  -> assembling
  -> source_ready
  -> transcoding
  -> ready
```

异常分支：

```text
recording/uploading -> interrupted -> uploading
recording/uploading -> abandoned
assembling          -> assemble_failed -> assembling (retry)
transcoding         -> transcode_failed -> transcoding (retry)
source_ready/ready  -> asset_missing (NAS 文件被外部删除)
```

状态含义：

- `created`：服务端已创建会话，尚未收到片段。
- `recording`：浏览器正在录制并上传。
- `uploading`：录制已结束，仍有片段等待上传。
- `interrupted`：客户端意外退出，服务端允许恢复缺失片段上传。
- `assembling`：片段齐全，worker 正在生成单一源文件。
- `source_ready`：源文件已通过 ffprobe 验证；此时才可计入打卡。
- `transcoding`：正在生成 720p MP4。
- `ready`：源文件和 720p 文件均可用。
- `abandoned`：用户明确放弃或长期未恢复。
- `asset_missing`：数据库记录存在，但一个或多个 NAS 文件缺失。源文件和压缩文件的缺失情况必须分别记录。

录制的主状态用于表达当前处理阶段，不能作为“源文件是否曾验证成功”的唯一依据。`source_validated_at` 一旦写入，除非源文件随后被检测为缺失或损坏，否则即使主状态进入 `transcoding` 或 `transcode_failed`，该录制仍满足阅读完成条件。

状态转换必须由服务端验证，客户端不得通过直接提交状态字符串跳过步骤。

### 7.2 分片上传协议

- 创建会话时服务端返回不可预测的 `recording_id` 和上传令牌/会话权限。
- 每个片段包含：`recording_id`、从 0 开始的 `sequence`、字节长度、SHA-256、MIME 类型、客户端片段开始/结束时间。
- 上传接口必须幂等。相同序号和相同哈希重复上传返回成功；相同序号但哈希不同返回冲突。
- 服务端先写 `.part`，完成长度和哈希校验后原子改名，数据库再标记已接收。
- 完成录制接口提交最终片段数、客户端总时长和最后序号。
- 服务端返回缺失序号列表；客户端补齐后再次完成。
- 单片建议控制在约 1–8 MB。服务端反向代理请求体限制应至少允许 16 MB。

### 7.3 浏览器恢复

IndexedDB 保存：

- 录制会话元数据。
- 尚未获得服务端确认的 Blob。
- 最终片段数和完成请求是否已发送。
- 服务端最后一次确认状态。

页面重开后先查询服务端已接收序号，再仅补传本地缺失片段。服务端已有的片段不重复占用存储。

如果浏览器丢失了服务端缺少的片段，应用将录制标记为“无法完整恢复”，但保留已上传片段供管理员诊断；不得伪装为完整视频。

### 7.4 源视频组装

不同 Safari/iOS 版本产生的 MediaRecorder 片段封装可能不同，不能假设所有 Blob 都能简单字节拼接。组装器必须：

1. 记录首片 MIME、编码信息和每片大小。
2. 针对实际格式选择 fMP4 拼接、FFmpeg concat/remux 或逐片解封装后拼接。
3. 输出一个 `source.<ext>.part` 临时文件。
4. 使用 ffprobe 验证容器、视频轨、音频轨、时长和可解码性。
5. 验证成功后原子改名为正式源文件。

必须用真机生成的片段建立测试样本，覆盖当前支持的最低 iOS/iPadOS 版本。实现者不得仅用桌面 Chrome 的 WebM 样本验证。

### 7.5 720p 转码规则

目标输出：

- 容器：MP4。
- 视频：H.264，`yuv420p`，宽度不超过 1280、高度不超过 720，保持纵横比，不放大低分辨率源。
- 音频：AAC-LC，单/双声道均可，目标 96–128 kbps。
- 帧率：保留源帧率但最高 30fps。
- Fast Start：`moov` 原子前置，便于 iOS 播放和下载。
- 文件名：人类可读名称通过下载响应提供，磁盘内部使用稳定 ID，避免中文路径兼容问题。

压缩目标以清晰阅读和手机空间平衡为主，建议平均视频码率约 1.2–1.8 Mbps。预计 10 分钟约 100–150 MB，实际大小取决于画面运动和编码器。

硬件路径使用 DS918+ Intel 核显的 VAAPI，并挂载 `/dev/dri`。启动时运行一次能力检测；若设备、权限或编码器不可用，自动切换软件 `libx264`。软件回退时仍限制单任务并发，并在管理页显示“软件转码，速度可能较慢”。

转码完成后必须再次 ffprobe，并进行至少首、中、尾三个位置的解码抽检。只有通过后才能原子替换正式 `720p.mp4`。

## 8. 单词解析与 TTS 设计

### 8.1 解析流水线

```text
原始输入
  -> 文件类型解析 / OCR
  -> 候选行与来源位置
  -> 本地规则分词与清理
  -> 可选 MiMo 文本建议
  -> 用户确认
  -> 固化单词本版本
  -> 查缓存 / 生成缺失 TTS
```

解析器必须保留原始输入，不把 AI 输出当作事实。AI 接口不可用时，图片 OCR 和本地文档解析仍应工作。

### 8.2 MiMo 文本模型用途

OpenCode Go 可配置接口用于：

- 从杂乱 OCR 文本中建议哪些行像英文单词或短语。
- 提示可能的编号、页眉、页脚和无关文本。
- 标记疑似拼写错误并给出建议。

它不得：

- 未经用户确认自动修改拼写。
- 生成孩子需要背诵但原文件中不存在的新词。
- 成为打开或编辑单词本的硬依赖。

### 8.3 TTS 提供商接口

后端定义统一适配器，例如：

```text
synthesize(text, locale, voice, speed, output_format) -> audio bytes + metadata
healthcheck() -> provider/model/availability
```

默认实现为 MiMo V2.5 TTS。由于 MiMo 的具体模型 ID、音色名和 API 路径可能更新，相关值必须从管理页或环境变量配置，不能散落硬编码在业务逻辑中。实现时应以接入当日的小米官方 TTS API 文档为准。

默认参数：

- locale：`en-US`。
- voice：选择一个清晰、自然、无明显地域混杂的美式音色，由管理员试听确认。
- speed：1.0，允许管理页在 0.8–1.2 范围调整。
- 输出：统一规范化为 MP3，必要时由 FFmpeg 转换。
- 单词前后不自动朗读序号或例句。

### 8.4 音频缓存

缓存键由以下字段的规范化 JSON 计算 SHA-256：

- provider 名称和 API 版本。
- model ID、voice ID、locale、speed。
- 单词/短语规范化文本。
- 输出格式和音频规范化版本。

相同缓存键只生成一次。数据库记录原始响应元数据、创建时间、时长、文件哈希和失败次数，不保存 API Key。

修改音色、语速或模型会产生新缓存，不覆盖旧缓存，从而保证历史默写仍可复现。

### 8.5 TTS 失败处理

- 网络错误、限流和 5xx 使用指数退避重试，最多自动重试 5 次。
- 认证失败或余额/套餐问题不做无限重试，直接在管理页提示配置错误。
- 任务失败不阻止编辑单词本。
- 已缓存音频始终可播放，不受当前 API 可用性影响。
- 默写开始前显示缺失音频数量；当前项没有音频时不得记为已播放。

## 10. 统计口径

### 10.1 阅读完成

- 某语言某日“完成”条件：存在一个 `source_validated_at` 非空、`source_missing_at` 为空且被标记为正式打卡的视频。主状态处于 `transcoding` 或 `transcode_failed` 不影响该条件。
- 转码失败不取消阅读完成状态。
- 当天第一次通过源文件验证的录制自动成为正式视频；后续可手动替换。
- 阅读时长使用 ffprobe 验证后的 `verified_duration_ms`，不使用前端计时器作为最终统计。

### 10.2 完成率

默认每天都要求中文和英文各一次：

- 中文完成率 = 已完成中文的有效天数 / 有效统计天数。
- 英文完成率 = 已完成英文的有效天数 / 有效统计天数。
- 综合完成率 = 已完成的语言任务数 /（有效统计天数 × 2）。

历史完整周/月使用该周期全部自然日。当前周/月只统计到今天，不把未来日期算作未完成。孩子档案创建前的日期不进入分母。

### 10.3 连续天数

主页面的连续打卡要求同一天中文和英文均完成。另在详细统计中显示中文连续天数和英文连续天数。

- 当前连续天数：从今天向前计算；如果今天尚未结束且未完成，允许从昨天开始计算，避免白天过早归零。
- 最长连续天数：历史上连续满足双语完成条件的最大自然日区间。

### 10.4 默写正确率

- 单次正确率 = `correct / (correct + incorrect)`。
- `unscored` 不进入分母，结束会话前必须提示仍有未评分项。
- 周/月趋势按评分发生日期聚合所有已评分项目，而不是简单平均每场百分比，避免不同场次题量造成偏差。
- 单词累计错误次数按 `normalized_text` 聚合，但历史页面仍显示当次确认的原拼写。
- 修改历史评分后，汇总必须可重建并保持一致。

## 12. 安全和隐私

- 密码使用 Argon2id；不允许明文或可逆加密密码。
- API Key 通过 Docker secret 或仅后端可读的配置文件/环境变量注入。管理页读取时只返回掩码。
- Cookie 设置 `HttpOnly`、`Secure`、`SameSite=Lax`；修改请求使用 CSRF 防护或严格 Origin 校验。
- 登录、上传创建、TTS 测试等接口限速。
- 上传文件必须限制类型、大小和页数；解析器不得使用用户文件名直接构造磁盘路径。
- 下载接口检查会话，不提供永久公开 URL。
- PWA service worker 不缓存 API 响应、视频、原始导入文件或带认证数据的页面。
- 日志不得包含 API Key、密码、完整 Cookie、完整孩子视频路径或导入文件正文。
- 媒体和数据库只存 NAS，不上传到除用户配置的 MiMo API 以外的第三方。调用 MiMo 文本模型时只发送必要的候选文本，不发送视频或整份原始文档。

## 13. 后台任务可靠性

- worker 使用带租约的任务锁。进程崩溃后，超过锁超时的任务可重新领取。
- 每个任务必须幂等：重复执行不能产生两个正式输出或重复统计。
- 所有大文件先写临时文件，校验后原子改名。
- 转码并发固定为 1；TTS 可并发 2；OCR 默认并发 1，均可配置。
- 任务记录进度百分比和稳定错误码。
- 自动重试只用于预期的临时错误；格式损坏、认证失败等永久错误需人工处理。
- 已完成源视频不因 720p 转码失败、数据库统计重建失败或外部 API 失败而被删除。

## 14. 临时数据与永久保留策略

- 已验证的源视频：永久保留，不自动删除。
- 720p 视频：默认永久保留，允许以后增加明确的人工清理功能；第一版不自动删除。
- TTS 音频：永久缓存。
- 导入原文件：默认保留，便于追溯；管理页显示占用。
- 完成组装后且源文件校验成功的片段：保留 24 小时后清理，以便短期诊断。
- 用户明确放弃或超过 7 天未恢复的未完成片段：在后台清理前先标记为 `abandoned`，管理页可见。
- SQLite 在线备份：建议每天一次，至少保留最近 7 个日备份和 4 个周备份；实际 NAS Hyper Backup 策略由用户维护。

“永久保留”表示应用不执行自动过期删除，不能防止磁盘故障、用户通过 File Station 手动删除或备份策略错误。因此必须建议将 `/data` 目录纳入群晖 Hyper Backup。

## 15. 可观测性与管理

- 健康检查分别覆盖 HTTP、SQLite 可写、媒体目录可写、ffmpeg/ffprobe 可执行、VAAPI 可用性。
- 管理页显示最近任务、平均转码耗时、失败原因、磁盘剩余空间。
- 磁盘剩余空间低于配置阈值（默认 20 GB）时禁止开始新录制，并给出明确提示；已上传内容仍允许完成组装。
- 日志按大小轮转，默认保留 14 天。
- 为每个录制、导入、TTS 和后台任务使用关联 ID，便于排查但不暴露密钥。

### 16.1 单元测试

### 16.2 后端集成测试

- 分片乱序、重复、缺失、哈希冲突和完成补传。
- worker 崩溃后任务恢复。
- 源视频组装、ffprobe 验证、硬件转码模拟失败后软件回退。
- 图片、文本 PDF、扫描 PDF、DOCX、XLSX 的导入样本。

### 16.4 真机矩阵

至少覆盖：

- 一台当前主力 iPhone + 最新正式 iOS。
- 一台当前主力 iPad + 最新正式 iPadOS。
- Safari 普通标签页。
- 添加到主屏幕后的 PWA。
- 家庭 Wi-Fi 和蜂窝/外部网络各一次。
- 前置摄像头、后置摄像头、横屏和竖屏。
- 连续 15 分钟录制。
- 录制中短暂断网 30 秒、2 分钟后恢复。
- 上传完成前关闭页面，再重新打开恢复。

最低支持的 iOS/iPadOS 版本应由开发开始时根据家庭实际设备确定，并在 README 中写死；未真机测试的版本不得宣称支持。

## 17. 验收标准

### 17.1 阅读视频

1. iPhone 和 iPad 均能在 HTTPS 页面授权摄像头/麦克风并开始录制。
2. 在正常网络下连续录制 15 分钟，手机不会生成一份需要手动删除的系统相册源视频。
3. 片段上传后从 IndexedDB 清理，本地待上传占用可见。
4. 短暂断网后恢复，所有仍在本地的片段自动补传，不产生重复源视频。
5. 服务端只在片段齐全且源文件验证成功后标记打卡完成。
6. NAS 保留可播放、有画面和声音、时长与录制接近的源视频。
7. 自动生成最高 720p、H.264/AAC、可由 iOS 播放和保存到相册的 MP4。
8. 10 分钟典型阅读视频的压缩版目标约 100–150 MB；若超过 200 MB，管理页应能看到实际码率以便调优。
9. 同日多次录制不会覆盖源文件，并可切换正式打卡。
10. 硬件编码不可用时任务自动回退软件编码，并保留源视频。

### 17.2 单词默写

1. 图片、粘贴、XLSX、DOCX、文本 PDF 和扫描 PDF 均能产生可编辑候选清单。
2. OCR 或 AI 不确定内容必须呈现给用户确认，不能静默修改。
3. 已确认单词可生成并试听标准美式 MiMo TTS。
4. 相同文本和相同 TTS 配置重复使用时命中 NAS 缓存，不再次调用 API。
5. 默写支持固定导入顺序和一次会话内固定的随机顺序。
6. 默认隐藏答案，播放、重播、显示答案、对错均由家长手动触发。
7. 刷新页面可恢复当前默写进度、顺序、播放次数和已评分结果。
8. TTS 暂时不可用时，已缓存词仍能正常默写。

### 17.3 统计

1. 正式源视频验证成功后，中文/英文日历和时长统计正确更新。
2. 当前周期不把未来日期算作缺卡。
3. 双语连续天数、单语言连续天数和最长连续天数符合定义。
4. 默写修改评分后，单场、周/月趋势和错词次数同步更新。
5. 错词可生成新的复习列表且不破坏历史记录。

### 17.4 运维和安全

1. 整套服务可通过 Docker Compose 在 DS918+ 启动和升级。
2. 重启 app 或 worker 不丢失已确认片段、任务和默写进度。
3. API Key 不出现在前端、API 普通响应或日志中。
4. 未登录用户不能查看、下载视频或音频。
5. 管理员可查看磁盘用量、失败任务并进行安全重试。

## 18. 开发交付物要求

后续编码交付至少应包含：

- 单仓库前端、后端、worker 源码。
- 数据库迁移脚本，不允许只靠运行时隐式建表。
- Dockerfile、`compose.yaml` 和 `.env.example`，示例中不得包含真实密钥。
- 群晖部署 README，包括共享目录权限、`/dev/dri` 映射、反向代理需要的最大请求体和超时说明。
- OpenAPI 文档。
- 自动化测试和真机测试清单。
- 一组不含家庭隐私的短视频片段夹具和文档导入夹具。
- 数据备份与恢复说明。
- 已知 iOS 限制和故障排查说明。

## 19. 建议的交付阶段

以下仅用于控制风险，不改变最终范围：

1. 基础骨架、认证、SQLite、任务队列和群晖卷挂载。
2. 真机录制、分片上传、恢复和源视频组装；这是最高风险阶段，应最先做技术验证。
3. 720p 转码、下载保存、视频库和阅读统计。
4. 文档导入、OCR、单词确认和版本管理。
5. MiMo TTS 适配器、缓存和默写会话。
6. 默写统计、错词复习、管理与备份。
7. iPhone/iPad 真机回归、断网测试和 DS918+ 性能调优。

在大规模开发前，应先用目标 iPhone/iPad 做一个最小技术探针，验证 Safari MediaRecorder 输出格式、分片可恢复性、源视频组装和 iOS 保存 720p MP4 的完整链路。该探针通过后再扩展完整页面。

## 20. 明确不留作开发者自行猜测的决策

- 第一版是 PWA，不是原生 App。
- 中文和英文分别录制。
- 源视频与 720p 视频默认都保存在 NAS；源视频不自动删除。
- 720p 成片由用户手动保存到 iOS 相册。
- 首次成功录制自动成为正式打卡，后续录制不自动替换。
- 阅读打卡以完整源视频验证成功为准，不以转码成功为准。
- 每天默认要求中文、英文各一次；综合连续天数要求两项都完成。
- 默写由家长手动控制节奏和人工评分。
- 默写默认按导入顺序，可选择固定随机顺序。
- TTS 默认 MiMo 美式英语，音频永久缓存，提供商可替换。
- OpenCode Go 文本模型只作可选建议，不是核心流程硬依赖，也不能替代 TTS。
- 第一版一个管理员、一个默认孩子，数据模型为多孩子预留。
