# Configurable TTS Settings Design

## Goal

Let a parent configure text-to-speech in the web UI without rebuilding the NAS container, while retaining MiMo as the default and allowing future OpenAI-compatible TTS providers.

## Design

The backend stores one TTS configuration record in SQLite. It contains the selected protocol (`mimo` or `openai_compatible`), base URL, model, voice, and speed. The API key is encrypted with a Fernet key stored only in the NAS data directory; reads expose only a boolean and masked suffix.

The settings API provides a read/update configuration endpoint and a test endpoint. Updates accept an empty API key to retain the existing secret. The TTS worker selects the adapter from the saved protocol, and the cache key includes protocol, endpoint, model, voice, speed, and normalized text so audio generated by different providers is never reused incorrectly.

The responsive settings page presents a provider selector, editable protocol fields, a password field that never pre-populates, save action, and a test-word action. MiMo defaults remain `https://api.xiaomimimo.com/v1`, `mimo-v2.5-tts`, and `Chloe`; OpenAI-compatible mode accepts a user supplied `/v1` base URL and model.

## Safety And Failure Handling

API keys never appear in GET responses, logs, test responses, or frontend state after a save. Validation rejects invalid URLs, unknown protocols, empty models/voices, and speeds outside 0.5 through 2.0. A failed test reports a stable error code without leaking upstream response content.

## Verification

Backend tests cover secret masking, updates retaining a secret, selecting both adapters, and cache separation. Frontend tests cover provider choice, fields, and no secret rendering. The complete Docker stack is rebuilt and served locally on port 8011.
